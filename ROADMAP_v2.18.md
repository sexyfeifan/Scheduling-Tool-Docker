# v2.18 升级路线图

## 目标

`v2.18` 的重点不是继续堆功能，而是把项目从“可以用”升级到“更稳定、更容易维护、更容易持续发布”。

本轮建议优先完成以下四个方向：

1. 拆分前后端超大单文件
2. 补齐自动化测试
3. 完善健康检查与部署质量
4. 增加高价值体验功能

---

## P0：必须优先完成

### 1. 拆分后端结构

当前后端核心逻辑集中在 `server/server.js`，建议拆分为：

- `server/app.js`
- `server/routes/schedules.js`
- `server/routes/settings.js`
- `server/routes/backup.js`
- `server/services/storage.js`
- `server/services/backup.js`
- `server/utils/normalize.js`
- `server/utils/security.js`

验收标准：

- 路由职责清晰
- 存储逻辑与 HTTP 逻辑分离
- 备份恢复逻辑可独立测试
- `server.js` 只负责启动服务

### 2. 拆分前端结构

当前前端主逻辑集中在 `client/js/main.js`，建议至少拆分为：

- `client/js/api.js`
- `client/js/state.js`
- `client/js/schedule.js`
- `client/js/settings.js`
- `client/js/backup.js`
- `client/js/export-image.js`
- `client/js/modal-project.js`
- `client/js/mobile.js`
- `client/js/utils/date.js`

验收标准：

- API 请求逻辑单独管理
- 排期渲染和业务操作分离
- 备份恢复、导入导出独立模块化
- 便于后续继续加功能

### 3. 补自动化测试

建议先覆盖最容易出事故的链路：

- 排期新增、修改、删除
- 设置保存与读取
- 备份创建、恢复
- 损坏文件自动从 `.bak` 恢复
- 旧格式备份兼容恢复

建议工具：

- 后端：`vitest` 或 `jest` + `supertest`
- 前端：先不强行补 UI 自动化，优先补核心纯函数与导入逻辑

验收标准：

- 关键 API 至少有基础回归测试
- 备份恢复链路可一键验证
- 每次发布前可自动跑测试

### 4. 健康检查与运行状态

新增：

- `/api/health`
- Docker `HEALTHCHECK`
- 启动日志中输出 `DATA_DIR` 和 `BACKUP_DIR`

验收标准：

- Docker / NAS 可直观看到容器健康状态
- 用户出问题时能快速判断是服务挂了还是数据异常

## P1：建议这一版顺手做

### 1. 撤销能力

高价值功能，建议至少实现一项：

- 删除项目后可撤销
- 拖拽移动后可撤销
- 恢复前显示“将覆盖当前数据”的摘要

### 2. 数据版本与迁移

给数据文件加 schema version，例如：

- `settings.json`
- `schedules.json`

增加迁移逻辑：

- `v1 -> v2`
- `v2 -> v3`

价值：

- 后续新增字段不会再靠临时兼容兜底
- 升级历史版本更稳

### 3. 备份体系增强

建议补充：

- 备份备注字段
- 自动保留最近 N 份
- 备份文件校验
- 恢复历史记录

### 4. 导入恢复体验优化

建议补：

- 导入前预览备份摘要
- 显示包含的日期范围和项目数量
- 恢复完成后显示恢复结果摘要

---

## P2：中期升级方向

### 1. 搜索和筛选

建议支持：

- 按日期范围筛选
- 按项目名搜索
- 按导演 / 摄影 / 类型筛选

### 2. 模板能力

建议支持：

- 常用项目模板
- 整周模板复制
- 固定班底一键套用

### 3. 分享与只读展示增强

建议支持：

- 指定周只读分享
- 隐藏部分字段
- 更适合投屏和对外展示的只读模式

### 4. 权限能力

如果后面要多人协作，建议再考虑：

- 编辑密码
- 只读访问密码
- 管理操作二次确认

---

## 推荐实施顺序

### 第一阶段

- 拆后端
- 补后端测试
- 增加健康检查

### 第二阶段

- 拆前端
- 规范 API / 状态 / 视图分层
- 优化导入恢复体验

### 第三阶段

- 增加撤销能力
- 增加模板、筛选、展示增强

---

## v2.18 建议发布说明

如果按这个路线推进，`v2.18` 最适合作为“稳定性与工程化升级版”发布，而不是只强调新功能。

建议对外描述重点：

- 数据持久化更稳
- 备份恢复更可靠
- Docker 多架构发布
- 项目结构更清晰，后续维护和迭代更安全

---

## 下一步建议

最推荐直接开始做的第一项：

1. 拆 `server/server.js`
2. 加 `/api/health`
3. 建立第一批后端回归测试

这一组投入不大，但收益很高，而且能直接为 `v2.18` 打基础。
