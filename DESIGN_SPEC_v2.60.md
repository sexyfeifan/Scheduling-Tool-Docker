## 2. 项目概述与目标

### 2.1 项目背景

罐头场通告排期系统是一个**通告单排片管理工具**，用于管理摄影棚的拍摄排期。系统核心功能包括：
- 按周视图展示每日排期项目
- 项目卡片的拖拽排序和编辑
- 自定义角色字段（导演、摄影师、制片等）
- 撤销/重做操作
- 数据导出（CSV、图片）
- Webhook 推送

### 2.2 当前痛点

| 痛点 | 详细描述 |
|------|----------|
| 缺少跨周视角 | 周视图只能看 7 天，跨周项目被截断，无法总览月度排期 |
| 缺少人员维度 | 无法按人员查看排期，分配工作时不知道某人是否已有其他安排 |
| 缺少历史数据 | 无法回顾已过期的排期数据，无法进行统计分析 |
| 手动同步日历 | 排期更新后需要手动通知，手机日历无法自动同步 |
| 重复填写项目 | 每次创建项目要手动填十几个字段，重复劳动多 |
| 单条操作低效 | 需要同时处理多个项目时只能逐个操作 |

### 2.3 v2.60 目标

| 优先级 | 目标 | 对应功能 |
|--------|------|----------|
| P0（必须） | 用户能按人员维度查看排期和冲突 | 功能④ |
| P0（必须） | 排期数据能自动推送到手机日历 | 功能⑩ |
| P1（重要） | 能总览月度排期 | 功能①② |
| P1（重要） | 减少重复操作 | 功能⑥⑦⑧⑨ |
| P2（期望） | 手机端体验优化 | 功能⑬⑭ |
| P2（期望） | 智能提醒 | 功能⑪ |

### 2.4 硬性约束

| 编号 | 约束 | 违反后果 |
|------|------|----------|
| R1 | 周视图（主视图）不做任何改动 | 用户无法使用 → 项目失败 |
| R2 | 所有新增视图均为辅助视图 | 架构混乱 → 无法维护 |
| R3 | 不修改现有数据模型字段 | 现有数据丢失 → 不可逆 |
| R4 | 现有 19 个测试必须全部通过 | 回归缺陷 → 线上故障 |
| R5 | 所有改动在 `v2.60-dev` 分支 | 正式版被污染 → 用户受影响 |

## 5. 功能规格说明书

### 5.1 功能⑮：代码架构优化

#### 5.1.1 用户故事

作为开发者，我需要后端代码结构清晰，以便后续高效添加新功能。

#### 5.1.2 当前问题

| 问题 | 详细描述 |
|------|----------|
| 路由层职责混杂 | `app.js` 中的路由虽然已拆分到 `routes/`，但 `schedules.js` 中包含了业务逻辑 |
| 中间件散落 | `requireAdminPassword` 等中间件定义在 `app.js` 中，应独立 |
| 缺少服务层 | 路由直接调用 `sqliteStore`，没有业务逻辑封装层 |
| 无数据访问抽象 | 如果将来换数据库，需要修改所有路由文件 |

#### 5.8.3 文本解析算法

```
输入："项目X 棚A 张导 李摄 视频"

解析步骤：
1. 按空格分割文本 → ["项目X", "棚A", "张导", "李摄", "视频"]
2. 从设置中加载已知列表：
   - commonLocations: ["棚A", "棚B", "外景"]
   - commonDirectors: ["张导", "李导"]
   - commonPhotographers: ["王摄", "赵摄", "李摄"]
   - 项目类型关键词: ["视频", "平面", "试做", "外拍", "直播"]
3. 遍历每个词，按优先级匹配：
   a. 匹配类型关键词 → type = "视频"
   b. 匹配场地列表 → location = "棚A"
   c. 匹配导演列表 → director = "张导"
   d. 匹配摄影师列表 → photographer = "李摄"
   e. 匹配制片列表 → production
   f. 匹配录音列表 → audio
   g. 匹配研发列表 → rd
   h. 匹配运营列表 → operational
   i. 匹配商务列表 → business
   j. 未匹配的词 → 拼接到 name
4. 输出解析结果对象

解析结果：
{
  name: "项目X",
  location: "棚A",
  director: "张导",
  photographer: "李摄",
  type: "视频",
  date: "2026-06-23",
  startTime: "09:00"
}
```

#### 5.12.2 数据流设计

```
┌──────────────────┐     HTTP GET      ┌──────────────────┐
│   排片系统        │ ──────────────→  │   日历客户端      │
│                  │   /api/calendar   │                  │

```

#### 5.12.3 iCal 格式规格

**生成的 .ics 文件示例**：

```
BEGIN:VCALENDAR
VERSION:2.0
PRODID:-//罐头场//排期系统//CN
CALSCALE:GREGORIAN
METHOD:PUBLISH
X-WR-CALNAME:罐头场排期
X-WR-TIMEZONE:Asia/Shanghai
BEGIN:VEVENT
DTSTART:20260623T090000
DTEND:20260623T110000
SUMMARY:项目A（视频）
LOCATION:棚A
DESCRIPTION:导演：张导\n摄影：李摄\n制片：王制片\n状态：已确认
STATUS:CONFIRMED
UID:schedule_2026-06-23_projectA@stel
lux
LAST-MODIFIED:20260623T080000Z
END:VEVENT
BEGIN:VEVENT
DTSTART:20260624T100000
DTEND:20260624T120000
SUMMARY:项目B（外拍）
LOCATION:外景
DESCRIPTION:导演：李导\n摄影：赵摄\n状态：待确认
STATUS:TENTATIVE
UID:schedule_2026-06-24_projectB@scheduling-tool
LAST-MODIFIED:20260624T070000Z
END:VEVENT
END:VCALENDAR
```

**字段映射规则**：

| 日历字段 | 数据来源 | 说明 |
|----------|----------|------|
| DTSTART | `{date}T{startTime}:00` | 如果 startTime 为空，默认 09:00 |
| DTEND | DTSTART + 2小时 | 默认事件时长 2 小时 |
| SUMMARY | `{name}（{type}）` | 项目名称 + 类型 |
| LOCATION | `location` 字段 | 拍摄场地 |
| DESCRIPTION | 所有人员字段拼接 | 格式：`职能：人名\n...` |
| STATUS | `status` 字段映射 | 待确认→TENTATIVE, 已确认→CONFIRMED, 已完成→CONFIRMED, 取消→CANCELLED |
| UID | `schedule_{date}_{name}@scheduling-tool` | 唯一标识，用于日历客户端去重 |
| LAST-MODIFIED | 当前时间戳 | 每次生成时更新 |

**API 设计**：
